LangGraph Removal Migration Summary
==================================
Branch: refactor/remove-langgraph
Date: 2026-04-01

FINAL STATE
-----------

LangGraph has been fully removed from the runtime and architecture. The system
now uses:

- a plain LangChain tool-calling loop for Teams queries
- deterministic queued workflow executors for webhook-driven business actions
- Excel as the workflow source of truth
- Cosmos for app state, Teams/session continuity, and adaptive card state

This is no longer a graph-based orchestration system. It is a deterministic
workflow app with an LLM-assisted Teams query surface.


WHAT WAS COMPLETED
------------------

1. Removed LangGraph runtime and checkpointing
   - deleted graph/state/checkpointing code
   - removed LangGraph dependencies and related config
   - removed checkpoint-era Terraform variables and resources

2. Replaced graph orchestration with a plain runner
   - `src/onboarding_agent/agent/runner.py` now owns:
     - prompt construction
     - tool loop execution
     - bounded retry/tool-loop behavior
     - compact session-context continuity

3. Moved webhook flows to deterministic execution
   - new hire, promotion, pay increase, transfer in, rehire, DocuSign, and
     background-clearance paths now execute in code, not by LLM orchestration
   - workflow stage policies were centralized under:
     - `src/onboarding_agent/domains/onboard/policies.py`

4. Simplified workbook/integration structure
   - workbook access split into focused clients:
     - `graph_workbook.py`
     - `tracker_client.py`
     - `staff_roster_client.py`
   - Teams logic grouped under `integrations/teams/`
   - legacy compatibility layers and dead models removed

5. Split MCP tools by domain
   - tracker, roster, Teams, onboarding, email, and DocuSign tools now have
     clear boundaries
   - webhook executors call runtime/integration code directly
   - LLM-facing tools remain for interactive Teams use

6. Standardized workflow identity around composite keys
   - tracker lookups now support:
     - `email + work_location + job_title + status_change`
   - adaptive card state, roster actions, background-clearance updates, and
     DocuSign metadata now propagate the same identity fields

7. Improved DocuSign resiliency
   - outgoing envelopes include hidden custom fields for workflow identity
   - webhook parsing carries those fields back into deterministic updates
   - onboarding status reconciliation still exists as a fallback for missed
     completion callbacks

8. Reduced legacy/stale code and payload compatibility branches
   - old prompt-builder webhook paths removed
   - stale payload alias handling pruned after Power Automate contract updates
   - repo structure reduced to the current monolith Container App model


CURRENT ARCHITECTURAL POSITION
------------------------------

The app is intentionally one deployable Container App, but with clear internal
separation:

- deterministic workflow runtime
- tool-calling Teams agent
- typed integration clients
- persisted app/card/session state

That gives the repo a clean path to future extraction of roster or Excel-heavy
domains into a separate internal service if needed, without reintroducing
LangGraph.

IMPORTANT BEHAVIORAL NOTES
--------------------------

1. Workflow ownership
   - Excel tracker owns workflow/business progression
   - Cosmos owns app/card/session continuity
   - the LLM does not own workflow state

2. Agent responsibility
   - Teams queries: interpret, call tools, summarize
   - webhook business workflows: deterministic only

3. Card and tracker identity
   - email-only identity is no longer sufficient
   - status-sensitive flows now rely on composite identity propagation


FUTURE CONSIDERATIONS
---------------------

1. DocuSign and status tooling
   - continue moving offer-letter/status queries toward composite-aware
     envelope resolution first, then envelope-status inspection

2. Roster domain modularization
   - roster CRUD is the likely next area for a dedicated domain/service
     boundary if Excel functionality expands materially

3. Infrastructure split
   - long-term direction remains documented in `infra/FUTURE_INFRA.md`
   - foundation/app Terraform separation is still the preferred future model

4. Excel scaling
   - tracker now benefits from table-based access and identity-aware lookups,
     but large tracker growth still warrants further narrowing of reads and tool
     outputs
