LangGraph Removal Migration Summary
==================================
Branch: refactor/remove-langgraph
Date: 2026-03-31

FINAL STATE
-----------

LangGraph has been removed from the application runtime. The repo now runs a
plain LangChain-style agent loop with explicit external state, queue-backed
webhook handling, and bounded Teams conversation memory.

The migration is complete from an application, documentation, and Terraform
cleanup standpoint. Validation is green.


WHAT WAS COMPLETED
------------------

1. Replaced LangGraph runtime with `agent/runner.py`
   - Plain async agent loop: invoke model -> execute tools -> repeat -> stop
   - Explicit retry handling and loop bounds
   - MCP initialization moved into the runner
   - No graph topology, checkpoints, or graph-owned state remain

2. Removed LangGraph application code
   - Deleted:
     - `src/onboarding_agent/agent/graph.py`
     - `src/onboarding_agent/agent/nodes.py`
     - `src/onboarding_agent/agent/state.py`
     - `src/onboarding_agent/runtime/checkpointing.py`

3. Removed LangGraph dependencies and config
   - Removed LangGraph packages from `pyproject.toml`
   - Removed graph/checkpoint config from runtime config
   - Updated package metadata and imports

4. Reworked Teams runtime around plain session memory
   - Replaced graph-thread concepts with session-keyed chat history
   - Consolidated Teams memory/session handling under `integrations/teams/`
   - Split Teams code into clearer modules:
     - `bot.py`
     - `memory.py`
     - `runtime.py`
     - `proactive.py`
     - `messenger.py`
     - `mentions.py`
     - `reply.py`
     - `card_actions.py`

5. Simplified and reorganized integrations
   - Extracted Forms into `forms_client.py`
   - Split workbook logic into:
     - `graph_workbook.py`
     - `tracker_client.py`
     - `staff_roster_client.py`
   - Removed the compatibility `graph_client.py` facade after callers were migrated
   - Removed dead `models/` package and unused employee model

6. Split MCP tools by domain
   - Replaced the old catch-all `tools_graph.py` with:
     - `tools_tracker.py`
     - `tools_staff_roster.py`
     - `tools_teams.py`
   - Kept `tools_onboarding.py`, `tools_docusign.py`, and `tools_email.py`
     focused on their own domains

7. Cleaned up infra and docs for the LangChain architecture
   - Removed checkpoint-era Terraform variables/resources/outputs
   - Updated README, runbook, architecture docs, and proposal status
   - Current Terraform provisions:
     - app state container
     - conversation session container
     - queue-backed webhook job processing
     - Container App runtime only

8. Removed legacy Cosmos compatibility code
   - `state_store_cosmos.py` now reads only canonical `payload` documents
   - Old `value` / `legacy_value` document handling was removed

9. Added a DocuSign reconciliation fallback
   - Real DocuSign completion callbacks proved unreliable in testing
   - Synthetic `/webhook/docusign` posts worked, confirming app-side handling was correct
   - Added reconciliation on onboarding status lookup:
     - find the latest envelope for the employee
     - if DocuSign is `completed` and tracker is missing `Offer Letter Signed`
     - update the tracker and emit the completion card once
   - This makes missed DocuSign callbacks non-blocking

10. Added/update tests for the new architecture
    - Deleted LangGraph-specific tests
    - Added runner, Teams memory, and reconciliation coverage
    - Updated job, MCP, and staff-roster tests for the new structure

BEHAVIORAL NOTES
----------------

1. Workflow state ownership is now explicit
   - Excel tracker remains the business workflow source of truth
   - Cosmos holds app state and ephemeral Teams memory
   - the agent runtime is short-lived and stateless between invocations aside
     from saved chat history

2. Teams memory is bounded operationally
   - inference context is trimmed in the runner
   - conversation session state is ephemeral and TTL-backed

3. DocuSign completion is now resilient
   - webhook remains the fast path
   - status reconciliation is the safety net for missed completion callbacks


FUTURE CONSIDERATIONS
---------------------

1. Infrastructure split
   - Future direction is documented in `infra/FUTURE_INFRA.md`
   - Long-term target:
     - `infra/terraform/foundation/`
     - `infra/terraform/app/`
   - Foundation should eventually provision long-lived Azure resources from the
     ground up
   - App layer should own the frequently replaced deployable resources

2. Azure Bot provisioning
   - Future Terraform implementation should likely use `azapi` for Azure Bot
     resources
   - Plan around single-tenant or managed-identity-style bot configuration
   - Do not assume Entra app registration can be fully Terraform-managed if IT
     controls Graph permission approval

3. DocuSign webhook reliability
   - Account-level Connect / webhook delivery is still worth improving
   - Even if DocuSign delivery becomes reliable, keep the reconciliation path as
     a defensive safety net

4. Excel scaling work
   - Tracker access still reads worksheet `usedRange` and scans in Python
   - If tracker size grows materially, refactor toward more selective lookups
     and smaller tool responses
