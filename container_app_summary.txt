Container App Deployment - Current Summary
==========================================

Branch: feature/container-app-deployment
Date: 2026-03-29


Current hosted shape
--------------------

The app now runs as a single Azure Container App that hosts:

- the aiohttp web server
- the Teams bot endpoint
- the webhook endpoints
- the in-process Azure Queue worker
- the FastMCP subprocess used by the LangGraph agent

Durable external dependencies:

- Azure Queue Storage for webhook handoff
- Cosmos DB `state-records` for app/card/conversation state
- Cosmos DB `langgraph-checkpoints` for LangGraph checkpoint persistence
- Azure Container Registry for the image
- Log Analytics for logs


What is now working
-------------------

1. Hosted Teams bot
   - DMs work
   - channel mentions work
   - proactive card posting works

2. Durable webhook ingress
   - new-hire webhook enqueues work
   - DocuSign webhook enqueues work
   - background-clearance webhook enqueues work

3. Durable state
   - Teams conversation references stored in Cosmos
   - card state stored in Cosmos
   - LangGraph checkpoints stored in Cosmos

4. Teams card actions
   - welcome email button works
   - offer letter button works
   - card state updates after actions

5. Status queries
   - tracker stages returned correctly
   - DocuSign status no longer incorrectly falls back to "no envelope"
   - stage dates are formatted as `MM/DD/YYYY`

6. Background clearance flow
   - confirmation email path fixed
   - flow now runs deterministically rather than relying entirely on the agent


Important implementation notes
------------------------------

1. Webhooks now return immediately
   The webhook handlers return a small `200 OK` JSON response after queueing.
   This works better with Power Automate's regular `HTTP` action.

2. Queue-backed processing
   Long-running work is no longer tied to the request lifecycle.

3. MCP subprocess inherits full runtime config
   This was required so the tool process could access the same secrets and
   state-store configuration as the main app.

4. Teams card actions are backgrounded
   This avoids the Teams client timeout behavior where a successful action could
   still display "Something went wrong."

5. Real Teams package remains local-only
   - `teamsappPackage/` is ignored
   - `teamsappPackage.example/` is committed


Known tradeoffs
---------------

1. The app is intentionally still one deployable service
   This keeps the deployment surface small, but means the web server, queue
   worker, and MCP subprocess all share one container.

2. Some flows are still agent-driven
   New-hire and DocuSign work still depend on model/tool orchestration rather
   than being fully deterministic.

3. LangGraph checkpoints can grow quickly
   The checkpoint container stores graph execution history, not just final
   outputs. Retention may eventually need TTL.

4. Performance is mostly I/O-bound
   Most latency comes from LLM, Graph, DocuSign, Teams, and Cosmos calls rather
   than raw compute alone.


Current deployment workflow
---------------------------

1. Sync staff roster config if needed
   `scripts/sync_staff_rosters_to_tfvars.sh infra/terraform/container-app/terraform.tfvars`

2. Build and push image
   `scripts/build_and_push_container_app.sh <acr-name> <image-tag>`

3. Update `image_tag` in `terraform.tfvars`

4. Apply Terraform
   `scripts/deploy_container_app.sh infra/terraform/container-app/terraform.tfvars apply`

5. Package Teams app if bot host changed
   `scripts/package_teams_app.sh <container-app-fqdn>`


Recommended next steps
----------------------

1. Lightweight CI
   Add a GitHub Actions workflow for:
   - dependency install
   - Python compile check
   - unit tests
   - Docker build
   - Terraform fmt/check

2. Staging deploy automation
   Keep production manual for now, but automate image build and staging deploy
   on a protected branch.

3. Post-deploy smoke tests
   Script the current manual checks:
   - `/api/messages`
   - Teams DM
   - Teams channel mention
   - one new-hire webhook
   - one background-clearance webhook

4. Checkpoint retention
   Add TTL to `langgraph-checkpoints` if Cosmos growth becomes operationally
   noisy.

5. New-hire flow hardening only if needed
   Leave it agent-driven for now, but if tracker-row creation becomes flaky in a
   real pattern, harden just that critical write first.

6. Better release notes
   Capture image tag, Terraform outputs, and smoke-test result for each deploy
   so rollbacks are easier.


When to revisit architecture
----------------------------

Only consider splitting the service further if one of these becomes persistent:

- queue throughput becomes a real bottleneck
- Teams traffic and webhook traffic interfere with each other
- MCP subprocess startup or isolation becomes a recurring problem
- cost or scale requirements justify a separate worker/runtime service

For now, the current one-container deployment is the right complexity level.
